## 二、定向引用机制：各节点 Prompt 改造

### 核心设计原则

在每个节点的 System Prompt 开头，新增一段**「算法规则库定向读取指令」**，格式如下：

```
【算法规则库读取指令】
当前节点任务为：{节点任务描述}
请在读取算法规则库（@{算法规则库}）时，优先定向读取以下章节，忽略其余章节：
- 主要依据：{章节列表}
- 次要参考：{章节列表}
执行时严格按照上述章节的规则约束生成输出，禁止泛读全文后自行归纳。
```

### Node 6 — 产品战略侧写师

**改动类型**：新增定向读取指令 + 补充 `capable_of` 第4类 COSMO 关联

**Step 1**：在 Node 6 的 Prompt **最前面**插入：

```
【算法规则库读取指令】
当前节点任务为：生成 writing_policy（场景优先级、能力场景绑定、禁止配对、FAQ限制、Bullet槽位规则）
请在读取算法规则库时，优先定向读取以下章节，忽略其余章节：
- 主要依据：
  · 第一章 1.2节（COSMO四类核心关联逻辑：used_for_func / used_for_eve / used_for_aud / capable_of）
  · 第二章（三系统协同关系）
  · 第四章（五点描述优化规则）
  · 第十二章（Rufus幻觉风险规避协议）
- 次要参考：附录A（COSMO意图映射快查表）
```

**Step 2**：在 `capability_scene_bindings` 决策逻辑部分，补充第4类关联类型：

```
COSMO 能力场景绑定支持四类关联类型：
1. used_for_func（功能关联）：产品功能性的常识关联
   → 示例：主动降噪耳机 → 覆盖"隔音、安静、专注"节点
2. used_for_eve（事件关联）：产品与特定事件/场合的关联
   → 示例：充电宝 → "商旅出差""长途飞行""音乐节"
3. used_for_aud（受众关联）：产品与受众群体的常识匹配
   → 示例：明确写出"专为游戏玩家""通勤族首选""适合学生"
4. capable_of（能力推理）【v2.0新增】：对产品底层能力的常识推理
   → 示例：写"3,950mAh × 7 cells"而非"20000mAh"，提供底层能力证明
   → 触发场景：用户询问"虚标吗？""实际能充几次？"等验证类问题

生成 capability_scene_bindings 时，四类关联类型均须覆盖，缺少任何一类须标注原因。
```

### Node 7 — 创意策略（文案生成）

**改动类型**：新增定向读取指令 + 新增 Rule 3b（边界声明）+ 补充 A+ 字数下限约束

**Step 1**：在 Node 7 的 Prompt **最前面**插入：

```
【算法规则库读取指令】
当前节点任务为：生成 Title / Bullet Points / A+ 文案
请在读取算法规则库时，优先定向读取以下章节，忽略其余章节：
- 主要依据：
  · 第三章（标题优化规则）
  · 第四章（五点描述优化规则）
  · 第七章（图片与A+内容）
  · 第十三章（3C品类特殊规则与高频误区）
- 次要参考：
  · 附录A（COSMO意图映射快查表）
  · 附录B（Rufus幻觉风险规避速查表）

执行顺序：先读第三章生成标题 → 再读第四章生成Bullet → 再读第七章检查A+约束 → 最后读附录B做幻觉风险自查。
```

**Step 2**：在现有 5 条硬性约束规则后，**新增 Rule 3b 和 Rule 6**：

```
Rule 3b（边界声明强制）【v2.0新增】
每个 Listing 必须包含至少 1 条明确的边界声明句。
格式："Optimized for [场景A]; not engineered for [场景B]"
算法依据：有明确边界的描述被 Rufus 判定为高置信度事实，置信度高于模糊正面承诺。
违反此规则 → Rufus conflict_check 维度 -10分

Rule 6（A+ 内容字数下限）【v2.0新增】
A+ 页面可索引文本总字数不得低于 500 字（英文）。
统计范围：所有模块的正文文字（不含标题、alt text、表格表头）。
算法依据：Rufus RAG 抓取 A+ 内容时（P3级数据源），文本密度低于阈值会导致语义节点覆盖不足。
违反此规则 → Rufus fact_completeness 维度 -15分
```

  · 第十二章（Rufus幻觉风险规避协议——conflict_check评分依据）
- 次要参考：附录B（Rufus幻觉风险规避速查表）

评分时按章节顺序逐维度执行，禁止跳过任何维度。
```

**Step 2**：在 Module 8 评分体系的**价格竞争力维度（原10分）**下，补充具体判定子规则：

```
价格竞争力评分细则（共10分）【v2.0细化】
评分依据：产品定价相对品类中位数的偏差区间

定价区间                      | 得分
品类中位数的 85% - 110%（甜蜜区）| 10分（满分）
品类中位数的 70% - 85%          | 7分（偏低，可能引发质量疑虑）
品类中位数的 110% - 115%        | 6分（轻微溢价，尚可接受）
品类中位数的 115% 以上           | 0分（高溢价惩罚，Rufus推荐降级）
品类中位数的 70% 以下            | 3分（极低价，可能触发质量过滤）

备注：品类中位数从 Node 3 的 Price_Context 字段获取。
如果 Price_Context 未传入，此维度标注"数据缺失，跳过评分"，不计入总分分母。
```

**Step 3**：在评分完成后的输出中，新增以下两个检查项的评分输出：

  声明句原文: "[...]"

  A+正文字数: [N]字

  得分: [+15 / -15]（对应 fact_completeness 维度）
```

## 四、验证方法（更新完成后）

用同一个测试 ASIN 跑一次完整工作流，重点检查以下5个验证点：

1. **Node 3**：输出的 L1/L2/L3 分级列表中，L3词是否出现在 Search Terms 而非 Title
2. **Node 6**：`capability_scene_bindings` 输出中是否包含 `capable_of` 类型的条目
3. **Node 7**：Bullet 中是否出现"Optimized for X; not engineered for Y"格式的边界声明句
4. **Node 8**：评分输出中是否新增了 `boundary_declaration_check` 和 `aplus_word_count_check` 两个字段
5. **Node 8**：价格竞争力得分是否引用了 Node 3 传入的 Price_Context 数据（而非 LLM 自行估计）

如果第5项 Price_Context 字段当前未从 Node 3 传递到 Node 8，需在工作流连线中新增该字段的传递路径。
